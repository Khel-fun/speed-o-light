# Speed-O-Light

Speed-O-Light is a high-speed, web-based reaction game inspired by classic arcade "speed of light" and "whack-a-button" machines — with a fundamental difference: every session is **cryptographically fair and provably so**.

Traditional arcade games operate on trust. Speed-O-Light operates on proof. Using **Zero-Knowledge Proofs** and **Kurier-based on-chain verification**, the game guarantees that neither the server nor any party can manipulate light patterns, danger tile placement, or scores after a session begins. The outcome of every game is verifiable by anyone, without revealing any private gameplay inputs.

---

## Gameplay

A session lasts **60 seconds**. Every **440 ms**, one of **25 tiles** on a 5×5 grid lights up. Each lit tile is one of two kinds:

- **Reward light** — tap it to earn **5 XP**
- **Danger light** — tapping it ends the session immediately

Over a full session, **136 lights** glow in total, with **34 of them being danger tiles** (25%). The sequence of reward and danger tiles is **deterministically derived from a single random seed** using a Poseidon2-based PRF. The seed is generated by Kurier before the session starts, ensuring neither the player nor the server has any influence over what the grid will look like.

If the player survives the full 60 seconds without hitting a danger tile, the maximum theoretical score is **510 XP** (102 non-danger taps × 5 XP). After the session ends, the game state is submitted to the backend, which **generates a ZK proof** of the session's integrity and routes it through Kurier for verification. The verified result can optionally be **settled on-chain** to the `SpeedOLightState` contract deployed on Base Sepolia.

---

## Fairness Guarantees

Speed-O-Light provides cryptographic fairness at every step of the session lifecycle.

### 1. Pre-session: Committed Randomness

Before a session begins, the backend requests a **random seed from Kurier**. This seed is the sole input into the grid generation algorithm — neither the server nor the player can influence the resulting tile sequence. The seed is committed to the session record before any gameplay occurs.

### 2. Grid Determinism

The 136-tile sequence (positions and types) is generated by a **circuit-level algorithm** running inside the Noir proving circuit:

- **`gen_glow_sequence`**: A skip-over bijection over the 5×5 grid that ensures no two consecutive tiles land on the same position.
- **`gen_danger_sequence`**: A partial Fisher-Yates shuffle that selects 34 unique positions out of 136 as danger slots.

Because this logic lives inside the circuit, any claimed grid can be verified against the seed by simply re-running the circuit — no trust in the server required.

### 3. Post-session: Zero-Knowledge Proof

After a session ends, the backend generates a **UltraHonk ZK proof** (via `@aztec/bb.js` and `@noir-lang/noir_js`) that certifies:

- **CHECK 01** — All 136 tap entries in the player's session match the deterministic grid sequence derived from the committed seed.
- **CHECK 02** — The danger tap (the tile that ended the session, or a sentinel value `255` for a full run) references a valid danger tile at the correct position in the grid.

The proof attests to the validity of the session without revealing the player's raw tap data to any verifier.

### 4. Kurier-based Verification

The generated proof is submitted to **Kurier**, a ZK proof aggregation and verification network. Kurier independently verifies the UltraHonk proof and advances it through a lifecycle of states:

```
QUEUED → VALID → SUBMITTED → INCLUDED_IN_BLOCK → FINALIZED → AGGREGATION_PENDING → AGGREGATED
```

The backend polls Kurier every 2 minutes for status updates. Once a proof reaches `AGGREGATED`, it has been batched into an on-chain aggregated proof and is considered fully finalized.

### 5. Optional On-chain Settlement

The player can choose to publish their session result to the `SpeedOLightState` contract on **Base Sepolia (chain ID 84532)**. Settlement is authorized by an ECDSA signature produced by the server after Kurier confirms the proof is valid. This makes the score and proof reference permanently visible on-chain.

## The Proving Circuit

The circuit is the foundational component of the entire game architecture. It is the **source of truth** for what constitutes a valid session.

The main circuit (`packages/circuits/speed_o_light/speed_o_light_game_state/src/main.nr`) accepts:

```
fn main(seed: pub Field, tap_sequence: [tap; GLOW_COUNT], danger_tap: tap)
```

- `seed` — the public random seed committed before gameplay
- `tap_sequence` — the full 136-element array of the player's taps (private)
- `danger_tap` — the tap record for the tile that ended the session (private)

The circuit re-derives the entire grid from the seed and verifies that the player's tap record is consistent with it. Because `seed` is the only public input, the proof reveals nothing about the player's specific taps — only that they were valid.

The compiled circuit artifact lives at `packages/circuits/speed_o_light/target/speed_o_light_game_state.json` and is loaded at runtime by the proving infrastructure in `packages/proving_setup`.

## Getting Started

### Prerequisites

- [Node.js](https://nodejs.org/) v20+
- [pnpm](https://pnpm.io/) v9+
- [PostgreSQL](https://www.postgresql.org/) database (local or hosted, e.g. Supabase)
- [Redis](https://redis.io/) (local or hosted) for the BullMQ proof queue
- A [Kurier](https://kurier.xyz/) API key for ZK proof verification

---

### 1. Clone the Repository

```bash
git clone https://github.com/Khel-fun/speed-o-light.git
cd speed-o-light
```

### 2. Install Dependencies

```bash
pnpm install
```

### 3. Configure the Environment

Copy the example environment files and fill in your values:

```bash
cp apps/server/.env.example apps/server/.env
cp apps/web/.env.example apps/web/.env
```

**`apps/server/.env`**

```bash
# PostgreSQL connection (pooled and direct URLs required by Prisma)
DATABASE_URL=postgresql://<user>:<password>@<host>:6543/<db>?pgbouncer=true&sslmode=require
DIRECT_URL=postgresql://<user>:<password>@<host>:5432/<db>?sslmode=require

# CORS origin for the frontend
CORS_ORIGIN=http://localhost:3001

NODE_ENV=development

# Kurier ZK verification network
KURIER_URL=https://api-testnet.kurier.xyz/api/v1
KURIER_API=<your-kurier-api-key>
KURIER_CHAIN_ID=84532

# ECDSA private key used to sign settlement payloads
SIGNING_PRIVATE_KEY=<your-private-key>

# Redis connection for BullMQ proof queue
REDIS_URL=redis://localhost:6379

LOG_LEVEL=info
```

**`apps/web/.env`**

```bash
VITE_SERVER_URL=http://localhost:3000
VITE_CONTRACT_ADDRESS=0xAD37B135F992714E081ea53358C44ccaa9593384
VITE_CHAIN_ID=84532
```

### 4. Set Up the Database

Pull the Prisma schema from PostgreSQL database:

```bash
pnpm dlx prisma db pull
```

To generate the Prisma client after any schema changes:

```bash
pnpm run db:generate
```

To open Prisma Studio for inspecting database state:

```bash
pnpm run db:studio
```

### 5. Run Locally

Start all services (frontend + backend) in development mode:

```bash
pnpm run dev
```

Or start each individually:

```bash
pnpm run dev:web      # React frontend → http://localhost:3001
pnpm run dev:server   # Express API    → http://localhost:3000
```

The game is accessible at [http://localhost:3001](http://localhost:3001).
